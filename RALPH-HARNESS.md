# Ralph-Loop Harness - 无人机自主开发流程

> **版本**: 2.0
> **创建日期**: 2026-03-18
> **目标**: 通过AI多Agent协作完成无人机WiFi飞行控制开发
> **核心机制**: Hook拦截 + 流程监控

## 1. 系统概述

### 1.1 Agent角色定义

```
┌────────────────────────────────────────────────────────────────────────────┐
│                         Ralph-Loop Harness 架构                            │
├────────────────────────────────────────────────────────────────────────────┤
│                                                                            │
│   ┌──────────────┐    ┌──────────────┐    ┌──────────────┐                │
│   │   Leader     │───→│  Reviewer    │←───│   Worker     │                │
│   │   (编排)     │    │  (审核)      │    │  (实施)      │                │
│   └──────┬───────┘    └──────────────┘    └──────┬───────┘                │
│          │                                       │                         │
│          └─────────────────┬─────────────────────┘                         │
│                            │                                               │
│                   ┌─────────▼──────────┐                                   │
│                   │ Harness-Architect  │                                   │
│                   │   (流程监控)       │                                   │
│                   └────────────────────┘                                   │
│                                                                            │
└────────────────────────────────────────────────────────────────────────────┘
```

| Agent | 职责 | 不做什么 |
|-------|------|----------|
| **Leader** | 任务编排、计划制定、任务分配、进度跟踪 | 不写代码、不修复bug |
| **Reviewer** | 计划审核、代码审查、PR审查、质量把关 | 不实施代码、不分配任务 |
| **Worker** | 代码实施、测试编写、文档更新 | 不制定计划、不分配任务给其他agent |
| **Harness-Architect** | 流程监控、偏差纠正、机制优化 | 不直接实施代码 |

### 2.2 Harness-Architect监控点

Harness-Architect在以下节点进行监控和干预：

| 监控点 | 检查内容 | 干预动作 |
|--------|----------|----------|
| 计划制定后 | 计划是否符合user-intent | 要求重新制定 |
| 任务创建后 | 任务边界是否清晰 | 要求细化任务文档 |
| Worker执行中 | 是否偏离harness轨道 | 发出警告并纠正 |
| PR审查时 | 是否违反架构原则 | 要求修改 |
| 任务完成后 | 是否产生新技术债务 | 要求记录到tech-debt-tracker |

### 2.3 Hook机制 (v2.0新增)

Hook机制在关键节点拦截操作，验证是否符合Harness规范。

#### 2.3.1 Hook触发点

| Hook点 | 触发时机 | 检查内容 | 拦截动作 |
|--------|----------|----------|----------|
| **EnterWorktree** | Worker创建工作区前 | 分支命名规范、代码基础版本、任务绑定 | 拒绝非法工作区创建 |
| **Edit/Write前** | Worker修改文件前 | 文件是否在任务清单范围内、是否引入未声明依赖 | 阻止越界修改 |
| **合并到main前** | PR合并前最终检查 | 流程合规性、技术债务记录、测试通过状态 | 阻止违规合并 |

#### 2.3.2 Hook检查规则

**EnterWorktree Hook:**
- 检查分支名是否符合规范: `task-{NNN}-{description}`
- 检查是否基于正确的commit创建
- 检查Worker是否已读取任务文档

**Edit/Write Hook:**
- 检查待修改文件是否在 `task-*.md` 的 `相关文件` 列表中
- 检查是否引入未在 `依赖关系` 中声明的新依赖
- 检查是否修改与任务无关的文件

**合并前Hook:**
- 检查是否经过Reviewer审批
- 检查是否产生新的技术债务（需记录到tech-debt-tracker.md）
- 检查测试是否通过

#### 2.3.3 偏差处理流程

**Worker发现任务清单问题:**

- **简单问题**（如缺少头文件）→ Worker自行修复，PR中注明，通知Leader更新任务模板
- **复杂问题**（如设计缺陷、遗漏依赖）→ Worker停止工作，上报Leader，Leader修订任务清单，Reviewer重新审核，Worker重新领取任务

**Harness-Architect检测到偏离:**

1. 分析根本原因
2. 文档化到 `docs/harness-deviation.md`
3. 通知相关Agent纠正
4. 更新Harness文档防止复发
5. 严重偏离时暂停本轮循环

## 3. 任务规范

### 3.1 任务文档模板 (v2.0)

每个任务必须写入 `docs/exec-plans/active/task-{序号}-{描述}.md`：

```markdown
# Task {NNN}: {任务名称}

## 目标
一句话描述这个任务要完成什么。

## 背景上下文

### 相关代码
- 文件: `/path/to/file` - 关键内容说明

### 依赖关系
- 前置任务: Task {XXX}
- 外部依赖: [库/模块名]

### 硬件信息
- 涉及引脚: [引脚名]
- 外设: [外设名]
- 参考: `hardware-docs/pinout.md` 第 X 节

### 代码规范
- 遵循 STM32 HAL 风格
- 寄存器访问使用 volatile
- 中断安全要求

## 具体修改要求

### 文件 1: `/path/to/file1`
1. 实现 [功能A]
2. 添加 [功能B]
3. 修改 [功能C]

### 文件 2: `/path/to/file2`
1. [修改描述]

## 完成标准 (必须可验证)

- [ ] 标准 1: [具体验证方法]
- [ ] 标准 2: [具体验证方法]
- [ ] 测试通过: [如何测试]
- [ ] 代码审查通过

## 相关文件 (Hook范围检查用)
- `/path/to/file1`
- `/path/to/file2`
- `/path/to/test_file`

## 注意事项
- [潜在风险或特别说明]

## 验收清单 (Reviewer使用)
- [ ] 修改在任务范围内
- [ ] 无新引入的未声明依赖
- [ ] 测试覆盖修改点
- [ ] 无新的技术债务
```

**关键要求 (v2.0):**
- **相关文件** 必须完整列出，用于Hook范围检查
- **依赖关系** 必须声明，防止引入未预期依赖
- **完成标准** 必须可验证，Reviewer据此验收
- Worker无需自己调查，按清单直接实施

### 3.2 任务大小限制

- **单个任务最多修改 1-3 个文件**
- **每个任务应该在 30-60 分钟内完成**
- **任务之间尽量独立，减少依赖**

## 4. 技术债务管理

### 4.1 记录规则

**任何失败、警告、无法立即解决的问题都应记录到 `docs/exec-plans/tech-debt-tracker.md`**

格式：
```markdown
## 待处理

- [ ] {日期} - {具体描述} - 来源: {哪轮循环/哪个任务}

## 已解决

- [x] {日期} - {具体描述} - 解决: {日期} - 方案: {如何解决}
```

### 4.2 债务分类

| 类型 | 描述 | 处理优先级 |
|------|------|------------|
| **Blocker** | 阻塞后续开发 | P0 - 立即解决 |
| **Warning** | 可能引发问题 | P1 - 本轮或下轮解决 |
| **Tech Debt** | 代码债务 | P2 - 后续迭代处理 |
| **Nice to have** | 优化建议 | P3 - 有时间再处理 |

## 5. Agent工作流详细规范

### 5.1 Leader工作流

```
1. 读取上一轮progress.md和tech-debt-tracker.md
2. 根据user-intent确定本轮目标
3. 拆分任务，创建task-*.md文档到exec-plans/active/
4. 调用Reviewer审核计划
5. 审核通过后，创建Agent团队
6. 动态分配任务给Worker
7. 跟踪任务进度，Worker完成后立即分配新任务
8. 所有任务完成后，更新progress.md，准备下一轮
```

### 5.2 Reviewer工作流

```
1. 收到审核请求后，读取相关文档
2. 按照检查清单进行审核
3. 输出审核报告:
   - 状态: APPROVED / NEEDS_REVISION
   - 问题列表 (如有)
   - 建议改进 (如有)
4. 代码/PR审查时，确保:
   - 符合架构设计
   - 有适当测试
   - 不产生新的技术债务
```

**Reviewer检查清单**:
- [ ] 任务边界是否清晰?
- [ ] 完成标准是否可验证?
- [ ] 是否有遗漏的依赖?
- [ ] 是否考虑了错误处理?
- [ ] 是否符合代码规范?

### 5.3 Worker工作流

```
1. 从Leader接收任务文档
2. 读取任务需要的所有上下文
3. 在.worktree/创建独立分支
4. 实施代码修改
5. 编写/运行测试
6. 自测通过后，提交PR
7. 通知Reviewer审查
8. 根据反馈修改 (如有)
9. 合并到main，关闭worktree
10. 通知Leader任务完成
```

**Worker实施规范**:
- 使用 `EnterWorktree` 创建隔离工作区
- 每次提交前自测
- PR必须包含清晰的描述
- 合并后立即通知Leader

### 5.4 Harness-Architect工作流 (v2.0)

```
1. 每轮循环首先启动
2. 读取RALPH-HARNESS.md作为规范
3. 配置和维护Hook规则
4. 设计整体agent架构
5. 为每个任务推荐最合适的agent类型
6. 监控整个流程执行 (通过Hook拦截点)
7. 发现偏差时:
   - 分析根本原因
   - 文档化到 docs/harness-deviation.md
   - 制定纠正措施
   - 更新harness文档防止复发
   - 严重偏离时暂停本轮循环
8. 必要时向Leader提出流程改进建议
```

**Hook管理职责:**
- 配置Hook规则和检查逻辑
- 处理Hook触发时的验证
- 定期Review Hook规则，避免过度严格
- 文档化所有Hook拦截事件

**偏差检测清单:**

| 违规类型 | 严重程度 | 动作 |
|----------|----------|------|
| Worker询问澄清(任务不清) | 高 | 通知Leader重写任务 |
| 任务超过3个文件 | 高 | 要求拆分任务 |
| 完成标准不可验证 | 高 | 阻止直到修复 |
| Agent越权写代码 | 严重 | 立即干预 |
| PR中无测试 | 中 | 要求添加测试 |
| 技术债务未记录 | 中 | 要求记录债务 |
| 计划变更未文档化 | 中 | 要求更新计划 |

## 9. Sequential Thinking 规范 (v2.1)

> **目的**: 在关键决策点使用结构化思考，提高复杂嵌入式系统开发的准确性和安全性

### 9.1 核心价值

无人机飞控是**安全关键系统**，涉及:
- 硬件/软件边界条件复杂
- 实时控制算法需要精确数学推导
- 多传感器融合依赖时序和配置正确性
- 飞行器失控后果严重

**Sequential Thinking (ST)** 强制逐步推理、支持回溯修正、可验证思考过程，是复杂决策的"保险机制"。

### 9.2 触发条件 (何时使用ST)

每个Agent在执行任务前，必须根据以下清单判断是否调用ST：

#### Leader (team-orchestrator)

| 触发条件 | ST深度 | 说明 |
|----------|--------|------|
| 任务涉及硬件+软件边界 | 5-7步 | 如传感器驱动配置 |
| 任务有多个前置依赖 | 5-7步 | 需要分析依赖关系 |
| 任务涉及安全关键功能 | 7-10步 | 电机控制、PID、电源管理 |
| 需要选择算法方案 | 8-10步 | 滤波器选择、融合算法 |
| 任务拆分后涉及>2个模块 | 5-7步 | 跨模块协调 |

#### Worker - STM32 (stm32-embedded-engineer)

| 触发条件 | ST深度 | 说明 |
|----------|--------|------|
| 配置新外设 (I2C/SPI/UART) | 5-8步 | 时钟、引脚、时序验证 |
| 实现时序敏感代码 | 5-7步 | 中断、DMA、定时器 |
| 实现数学算法 | 8-10步 | PID、滤波器、AHRS姿态解算 |
| 修改中断服务程序 | 5-7步 | 中断安全、竞态条件 |
| 配置时钟树 | 5-7步 | 系统时钟、外设时钟 |
| 调试复杂硬件问题 | 直到解决 | 传感器异常、通信失败 |

#### Worker - ESP32 (esp32-c3-autonomous-engineer)

| 触发条件 | ST深度 | 说明 |
|----------|--------|------|
| 设计协议帧格式 | 5-7步 | 协议设计、边界情况 |
| 处理并发/竞态条件 | 5-8步 | 多任务、中断、资源竞争 |
| 实现状态机 | 5-7步 | 状态转换、异常处理 |
| WiFi/网络配置 | 5-7步 | 连接管理、重连策略 |
| 调试复杂网络问题 | 直到解决 | 连接异常、数据丢失 |

#### Reviewer (code-reviewer)

| 触发条件 | ST深度 | 说明 |
|----------|--------|------|
| 所有架构相关修改 | 5-7步 | 验证架构一致性 |
| 涉及硬件引脚的变更 | 5-7步 | 核对引脚定义 |
| 安全关键代码审查 | 7-10步 | 电机、PID、电源 |
| 算法实现审查 | 5-8步 | 数学正确性验证 |

#### Harness-Architect (harness-architect)

| 触发条件 | ST深度 | 说明 |
|----------|--------|------|
| 流程偏差根因分析 | 直到解决 | 系统性问题分析 |
| 设计新的Hook规则 | 5-8步 | 规则影响分析 |
| 多维度判断是否干预 | 5-7步 | 权衡干预成本 |

### 9.3 ST 输出要求

每次ST调用必须产生以下输出，并简要记录在任务文档中：

```markdown
## Sequential Thinking 记录

**触发原因**: [简述触发条件]
**思考摘要**: [1-2句话总结核心思考]
**决策结论**: [明确的行动决定]
**风险识别**: [识别的潜在风险，无则填"无"]
**验证建议**: [如何验证决策正确性]
```

### 9.4 实施示例

**示例1: STM32 Worker配置I2C**

```
触发: 配置新外设(I2C1 for ICM-42688-P)
深度: 6步

思考过程:
1. 确认I2C引脚: PB6(SCL), PB7(SDA) - 查pinout.md
2. 确认时钟配置: APB1 42MHz, 目标400kHz Fast Mode
3. 计算分频系数: 42MHz/(3+9+3)≈400kHz
4. 确认I2C地址: ICM-42688-P是0x68(AD0=GND)
5. 评估DMA需求: 数据量小，先轮询，后续优化
6. 风险评估: 时钟拉伸、总线仲裁 - 暂不处理，出问题再调试

输出:
- 摘要: I2C1配置已验证，引脚/时钟/地址正确，先轮询模式
- 结论: 使用HAL_I2C_Init+标准配置
- 风险: 需测试总线是否繁忙检测
- 验证: 用逻辑分析仪确认400kHz波形
```

**示例2: Leader拆分任务**

```
触发: 任务涉及传感器驱动+PID控制两个模块
深度: 7步

思考过程:
1. 分析依赖: 传感器必须先于PID工作
2. 确定边界: 传感器输出结构体作为契约
3. 评估风险: 数据时序不匹配可能导致PID震荡
4. 考虑方案A: 一个任务完成 → 太大，不符合1-3文件原则
5. 考虑方案B: 拆分为两个独立任务 → 需要明确接口
6. 设计接口: 定义sensor_data_t结构体，包含时间戳
7. 最终决策: 拆分为Task001(传感器)和Task002(PID)，Task001先执行

输出记录到任务文档
```

### 9.5 Reviewer 检查清单 (ST相关)

- [ ] Agent在需要时使用了ST (根据触发条件判断)
- [ ] ST输出已简要记录在任务文档
- [ ] 基于ST结论的决策合理
- [ ] 识别的风险有对应的缓解措施

### 9.6 豁免条件

以下情况**可以跳过**ST：

1. **紧急修复**: 飞行器紧急问题修复，可先实施后补ST记录
2. **简单变更**: 纯格式调整、注释修改、变量重命名
3. **重复任务**: 完全相同的配置（如复制I2C1配置到I2C2），且已验证过
4. **测试代码**: 临时测试脚本、调试打印

### 9.8 违规处理

**Reviewer发现未使用ST**:
- 非安全关键任务 → 提醒，要求后续遵守
- 安全关键任务 → 要求补充ST分析，重新审查

**Harness-Architect发现模式**:
- 记录到`docs/harness-deviation.md`
- 分析是触发条件不清晰还是Agent未遵守
- 必要时更新触发条件清单
