# Ralph-Loop Harness-Engineering 设计文档

> **版本**: 2.0
> **创建日期**: 2026-03-18
> **目标**: 通过 Hook 机制实现流程监控，防止 Ralph-loop 偏离轨道

---

## 1. 问题分析

### 1.1 上一版本的问题

| 问题 | 影响 | 根本原因 |
|------|------|----------|
| 缺乏流程监控 | 执行偏离计划不自知 | 没有专门监控 Harness 执行的机制 |
| 调查与实施混淆 | Worker 边调查边写代码，容易偏离 | 任务清单不够详细，Worker 需要自己调查 |
| 关键节点无检查 | 错误持续到合并后才暴露 | 没有设置流程检查点 |

### 1.2 设计目标

1. **分离关注点**：Leader 调查制定清单，Worker 只管实施
2. **前置检查**：关键节点设置 Hook，拦截偏离行为
3. **快速恢复**：简单问题 Worker 自治，复杂问题上报 Leader

### 2.1 Agent 职责矩阵

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         Ralph-Loop Harness v2.0                              │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│   ┌──────────────────────────────────────────────────────────────────┐      │
│   │                         Harness-Architect                        │      │
│   │                    (Hook拦截器 / 流程监控)                        │      │
│   │              ┌──────────────┬──────────────┬──────────────┐        │      │
│   │              │ EnterWorktree│  Edit/Write  │ Merge前检查  │        │      │
│   │              └──────────────┴──────────────┴──────────────┘        │      │
│   └──────────────────────────────────────────────────────────────────┘      │
│                                    ▲                                         │
│                                    │ 监控/纠正                                │
│   ┌──────────────┐    ┌──────────────┐    ┌──────────────┐                  │
│   │   Leader     │───→│  Reviewer    │←───│   Worker     │                  │
│   │  (调查+编排)  │    │  (审核)      │    │  (实施)      │                  │
│   └──────────────┘    └──────────────┘    └──────────────┘                  │
│          │                                          ▲                       │
│          └───────────────────┬──────────────────────┘                       │
│                              │ 任务清单                                        │
└──────────────────────────────┼──────────────────────────────────────────────┘
                               │
                               ▼
                   ┌───────────────────────┐
                   │   exec-plans/active/  │
                   │   - plan.md           │
                   │   - task-*.md         │
                   └───────────────────────┘
```

### 2.2 职责边界

| Agent | 职责 | 禁止事项 | 产出物 |
|-------|------|----------|--------|
| **Leader** | 调查上下文、制定计划、拆分任务、动态分配 | 不写代码、不修复 bug、不做架构决策 | plan.md, task-*.md |
| **Reviewer** | 审核计划、任务、代码、PR | 不实施代码、不分配任务、不覆盖 Architect 建议 | review-report.md |
| **Harness-Architect** | 监控流程、Hook 拦截、偏差纠正 | 不直接实施代码、不覆盖 Leader 决策 | agent-assignment.md, 偏差报告 |
| **Worker** | 读取任务、代码实施、测试验证 | 不制定计划、不分配任务 | PR, 测试报告 |

## 3. Hook 机制设计

### 3.1 Hook 触发点

| Hook 点 | 触发时机 | 检查内容 | 拦截动作 |
|---------|----------|----------|----------|
| **EnterWorktree** | Worker 创建工作区前 | 分支命名规范、代码基础版本、任务绑定正确性 | 拒绝非法工作区创建 |
| **Edit/Write 前** | Worker 修改文件前 | 文件是否在任务清单范围内、是否引入未声明依赖 | 阻止越界修改 |
| **合并到 main 前** | PR 合并前最终检查 | 流程合规性、技术债务记录、测试通过状态 | 阻止违规合并 |

### 3.2 Hook 实现方式

通过 Claude Code 的 Hook 机制（`settings.json`）实现：

```json
{
  "hooks": {
    "EnterWorktree": {
      "before": "harness-hook worktree-check"
    },
    "Edit": {
      "before": "harness-hook scope-check"
    },
    "Write": {
      "before": "harness-hook scope-check"
    },
    "Bash": {
      "before": "harness-hook merge-check"
    }
  }
}
```

### 3.3 Hook 检查规则

**EnterWorktree Hook:**
- 检查分支名是否符合规范: `task-{NNN}-{description}`
- 检查是否基于正确的 commit 创建
- 检查 Worker 是否已读取任务文档

**Edit/Write Hook:**
- 检查待修改文件是否在 `task-*.md` 的 `相关文件` 列表中
- 检查是否引入未在 `依赖关系` 中声明的新依赖
- 检查是否修改与任务无关的文件

**合并前 Hook:**
- 检查是否经过 Reviewer 审批
- 检查是否产生新的技术债务（需记录到 tech-debt-tracker.md）
- 检查测试是否通过

## 4. 执行流程

### 4.1 单轮循环流程

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           单轮 Ralph-Loop 流程                               │
└─────────────────────────────────────────────────────────────────────────────┘

Phase 1: 上下文收集
──────────────────────────────────────────────────────────────────────────────
Leader 读取:
  ├─ CLAUDE.md              (架构地图)
  ├─ RALPH-HARNESS.md       (Harness规范)
  ├─ docs/user-intent.md    (用户意图)
  ├─ docs/exec-plans/tech-debt-tracker.md  (技术债务)
  ├─ ralph-loop-session/progress.md        (上一轮进度)
  └─ 相关代码文件

Phase 2: 计划制定
──────────────────────────────────────────────────────────────────────────────
Leader 产出:
  ├─ docs/exec-plans/active/plan.md        (本轮计划)
  └─ docs/exec-plans/active/task-*.md      (任务清单)

Phase 3: 任务审核
──────────────────────────────────────────────────────────────────────────────
Reviewer 审核:
  ├─ 计划是否符合 user-intent
  ├─ 任务边界是否清晰
  ├─ 完成标准是否可验证
  ├─ 任务大小是否合适 (1-3文件)
  └─ 输出审核报告

Phase 4: 架构设计
──────────────────────────────────────────────────────────────────────────────
Harness-Architect:
  ├─ 读取所有 task-*.md
  ├─ 为每个任务推荐 Agent 类型
  ├─ 识别可并行任务
  └─ 产出 agent-assignment.md

Phase 5: 任务分配与执行
──────────────────────────────────────────────────────────────────────────────
Leader → Worker:
  ├─ 分配任务 (引用 task-*.md 路径)
  └─ 说明依赖关系和优先级

Worker 执行 (Hook 监控):
  ├─ [Hook] EnterWorktree 检查
  ├─ 读取任务文档
  ├─ 代码实施
  ├─ [Hook] Edit/Write 前范围检查
  ├─ 测试验证
  ├─ 创建 PR
  └─ 通知 Reviewer

Phase 6: PR 审查与合并
──────────────────────────────────────────────────────────────────────────────
Reviewer:
  ├─ 代码质量检查
  ├─ 测试验证
  └─ 技术债务检查

[Hook] 合并前检查:
  ├─ 是否通过 Reviewer 审批
  ├─ 技术债务是否记录
  └─ 测试是否通过

Leader:
  └─ 归档已完成任务

Phase 7: 下一轮准备
──────────────────────────────────────────────────────────────────────────────
Leader:
  ├─ 更新 ralph-loop-session/progress.md
  ├─ 记录本轮技术债务
  └─ 准备下一轮计划
```

### 4.2 偏差处理流程

**Worker 发现任务清单问题:**

```
Worker 发现:
  │
  ├── 简单问题 (如: 缺少头文件包含)
  │     ├─ Worker 自行修复
  │     ├─ 在 PR 描述中注明修复内容
  │     └─ 通知 Leader 更新任务模板
  │
  └── 复杂问题 (如: 设计缺陷、遗漏依赖)
        ├─ Worker 停止工作
        ├─ 上报 Leader
        ├─ Leader 修订任务清单
        ├─ Reviewer 重新审核
        └─ Worker 重新领取任务
```

**Harness-Architect 检测到偏离:**

```
Architect 检测到偏离:
  │
  ├─ 分析根本原因
  ├─ 文档化到 docs/harness-deviation.md
  ├─ 通知相关 Agent 纠正
  ├─ 更新 Harness 文档防止复发
  └─ 严重偏离时暂停本轮循环
```

## 6. 技术债务管理

### 6.1 记录规则

**任何失败、警告、无法立即解决的问题都应记录到 `docs/exec-plans/tech-debt-tracker.md`**

格式：
```markdown
## 待处理 (TODO)

- [ ] YYYY-MM-DD - {简要描述} - 来源: {任务/轮次} - 优先级: {P0/P1/P2/P3}

## 已解决 (Resolved)

- [x] YYYY-MM-DD - {简要描述} - 来源: {任务/轮次} - 解决: YYYY-MM-DD - 方案: {如何解决}
```

### 6.2 优先级定义

| 优先级 | 含义 | 处理时限 |
|--------|------|----------|
| **P0 - Blocker** | 阻塞后续开发 | 立即解决 |
| **P1 - Warning** | 可能引发问题 | 本轮或下轮解决 |
| **P2 - Tech Debt** | 代码债务 | 后续迭代处理 |
| **P3 - Nice to have** | 优化建议 | 有时间再处理 |

## 7. Agent 定义更新

### 7.1 Leader (team-orchestrator)

**核心职责**: 调查上下文 → 制定计划 → 拆分任务 → 动态分配

**新增要求**:
- 任务文档必须包含完整上下文，Worker 无需自己调查
- 任务边界必须清晰，Hook 可验证
- 每个任务必须指定相关文件列表（用于 Hook 范围检查）

### 7.2 Reviewer (code-reviewer)

**核心职责**: 审核计划 → 审核任务 → 审核代码 → 审核 PR

**新增要求**:
- 任务审核时验证完成标准是否可 Hook 检查
- PR 审核时检查是否引入未记录的技术债务

### 7.3 Harness-Architect

**核心职责**: 流程监控 → Hook 拦截 → 偏差纠正 → 持续改进

**新增要求**:
- 配置和维护 Hook 规则
- 处理 Hook 触发时的检查逻辑
- 文档化所有偏差并制定预防措施

### 7.4 Worker (各专业 Agent)

**核心职责**: 读取任务 → 代码实施 → 测试验证

**新增要求**:
- 严格在任务清单范围内工作
- 发现清单问题时按流程上报
- 配合 Hook 检查，不绕过验证

## 8. 成功指标

### 8.1 流程指标

| 指标 | 目标值 | 测量方法 |
|------|--------|----------|
| 任务清单一次性通过率 | >80% | Reviewer 审核通过比例 |
| Hook 拦截违规次数/轮 | <2 | Harness-Architect 记录 |
| Worker 上报问题率 | <20% | 任务修订次数/总任务数 |
| 技术债务增长率 | 递减 | 每轮新增债务数 |

### 8.2 代码指标

| 指标 | 目标值 |
|------|--------|
| 任务平均文件数 | 1-3 |
| 任务平均完成时间 | 30-60 分钟 |
| PR 一次性通过率 | >70% |

## 9. 风险与缓解

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|----------|
| Hook 过于严格导致效率降低 | 中 | 中 | 定期 Review Hook 规则，允许白名单 |
| Worker 绕过 Hook 检查 | 低 | 高 | Hook 在服务器端强制执行 |
| 任务清单过于详细导致 Leader 负担重 | 中 | 中 | 建立任务模板库，复用常见任务 |
| Harness-Architect 监控盲区 | 中 | 高 | 多维度检查（范围+依赖+流程） |
