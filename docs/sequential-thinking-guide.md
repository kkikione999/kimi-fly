## 什么是 Sequential Thinking?

Sequential Thinking 是一个 MCP 工具，强制 AI 进行逐步推理：

1. **逐步思考** - 每个 thought 建立在前一个基础上
2. **支持回溯** - 可以修正之前的思考
3. **分支探索** - 可以同时探索多个方案
4. **可验证** - 思考过程可追溯、可审查

### 为什么无人机飞控需要ST?

```
无人机飞控特点:
├── 安全关键: 失控后果严重
├── 硬件复杂: 多传感器、多接口
├── 实时要求: 时序敏感
├── 算法密集: PID、滤波、姿态解算
└── 边界复杂: 硬件/软件交互点多

ST的价值:
├── 减少配置错误 (I2C地址、SPI模式)
├── 确保算法正确 (数学推导可验证)
├── 识别边界条件 (异常处理完善)
└── 决策可追溯 (便于调试和审查)
```

## 触发条件详解

### Leader (team-orchestrator)

**触发场景1: 任务涉及硬件+软件边界**
```
示例: 编写ICM-42688-P IMU驱动

思考步骤:
1. 确认传感器接口: I2C1
2. 确认引脚分配: PB6/PB7
3. 确认时钟要求: 400kHz Fast Mode
4. 确认数据输出: 加速度+陀螺仪
5. 确定驱动边界: 向上提供原始数据/姿态数据?
6. 评估依赖: 是否需要DMP? 先原始数据
7. 最终决策: Task001-I2C配置, Task002-驱动框架, Task003-数据读取

输出记录:
- 拆分为3个顺序任务
- Task001完成后再执行Task002
- 接口定义为sensor_raw_data_t结构体
```

**触发场景2: 需要选择算法方案**
```
示例: 选择姿态解算算法

思考步骤:
1. 需求分析: 更新率1kHz, 资源受限
2. 方案A: 扩展卡尔曼滤波(EKF)
   - 优点: 精度高
   - 缺点: 计算量大, STM32F411可能吃力
3. 方案B: Mahony互补滤波
   - 优点: 计算量小, 适合1kHz
   - 缺点: 动态精度略低
4. 方案C: Madgwick梯度下降
   - 优点: 精度好, 计算适中
   - 缺点: 需要调参
5. 资源评估: F411 100MHz, 可用~50% CPU
6. 风险评估: 先实现Mahony, 后续可升级
7. 最终决策: 使用Mahony, 预留切换到Madgwick的接口

输出记录:
- 选择Mahony算法
- 预留算法切换接口
- 风险: 大机动时精度可能不足
```

### Worker - STM32 (stm32-embedded-engineer)

**触发场景: 配置新外设**
```
示例: 配置SPI3用于气压计LPS22HBTR

思考步骤:
1. 确认引脚:
   - SPI3_SCK: PC10
   - SPI3_MISO: PC11
   - SPI3_MOSI: PC12
   - CS: PA4 (软件控制)
   - 确认: 查pinout.md第3节

2. 时钟配置:
   - SPI3在APB1总线, 42MHz
   - LPS22HBTR最大10MHz
   - 分频: 42MHz/8 = 5.25MHz (安全)
   - 或者 /4 = 10.5MHz (略超, 用/8)

3. 模式配置:
   - CPOL: 0 (Idle低)
   - CPHA: 0 (第一个边沿采样)
   - 确认: 查LPS22HBTR datasheet图8-1

4. 数据格式:
   - 8位数据
   - MSB first
   - 无CRC

5. CS控制:
   - 软件控制, 每字节切换?
   - 查手册: 可以突发读取
   - 决定: 读取时拉低, 完成拉高

6. 错误处理:
   - 超时检测
   - 忙等待上限
   - DMA? 数据量小, 先不用

7. 验证计划:
   - 读取WHO_AM_I寄存器(0x0F)
   - 应返回0xB1

输出记录到任务文档:
```markdown
## Sequential Thinking Record
**Trigger**: 配置新外设(SPI3 for LPS22HBTR)
**Summary**: SPI3配置已验证: PC10/11/12, 5.25MHz, Mode0, 软件CS
**Decision**: 使用HAL_SPI_Init, 轮询模式, 先验证WHO_AM_I
**Risks**: 时钟略低于最大, 但更安全; 无DMA后续可能成为瓶颈
**Verification**: 读取0x0F寄存器应返回0xB1
```

**触发场景: 实现PID算法**
```
示例: 实现角度环PID控制器

思考步骤:
1. PID公式确认:
   - 位置式: u(t) = Kp*e + Ki*∫e + Kd*de/dt
   - 增量式: Δu = ...
   - 选择: 位置式, 直观易调试

2. 离散化:
   - 积分: 梯形法或矩形法
   - 微分: 后向差分
   - 采样率: 1kHz (1ms周期)

3. 抗积分饱和:
   - 必须实现! 电机输出有限
   - 方法: 积分限幅或条件积分
   - 选择: 条件积分(更平滑)

4. 微分滤波:
   - 原始微分对噪声敏感
   - 加一阶低通滤波
   - 截止频率: ~100Hz (10倍于控制带宽)

5. 输出限幅:
   - PWM范围: 1000-2000μs
   - PID输出映射到该范围
   - 限幅在映射前还是后? 映射前

6. 数据结构:
   - Kp, Ki, Kd: float
   - integral: float (持续累加)
   - prev_error: float
   - 是否线程安全? 1kHz定时器, 单线程, 安全

7. 参数整定策略:
   - 先P: 从小到大直到震荡
   - 加D: 抑制震荡
   - 加I: 消除静差
   - 初始值: P=1.0, I=0, D=0.1

8. 测试验证:
   - 阶跃响应测试
   - 检查超调量
   - 检查稳态误差
   - 检查噪声敏感度

输出记录:
```markdown
## Sequential Thinking Record
**Trigger**: 实现数学算法(PID控制器)
**Summary**: 位置式PID, 条件积分抗饱和, 微分加100Hz低通滤波
**Decision**: 先实现单轴, 验证后再扩展到三轴
**Risks**: 积分饱和导致失控; 微分噪声放大
**Verification**: 阶跃响应测试, 超调<20%, 稳态误差<1度
```
```

### Worker - ESP32 (esp32-c3-autonomous-engineer)

**触发场景: 设计协议帧格式**
```
示例: 设计无人机控制协议

思考步骤:
1. 需求分析:
   - 控制命令: throttle, pitch, roll, yaw
   - 数据类型: 16位有符号整数足够
   - 范围: -1000~+1000 (0.1度或0.1%)

2. 帧结构设计:
   - SOF: 0xAA 0x55 (避免与数据混淆)
   - CMD: 1字节 (命令类型)
   - LEN: 1字节 (payload长度)
   - PAYLOAD: N字节
   - CRC: 1字节 (CRC8)
   - EOF: 0x0D (可选, 用于调试)

3. 命令定义:
   - 0x01: 控制命令 (4x2字节 = 8字节payload)
   - 0x02: 心跳 (0字节payload)
   - 0x03: 遥测请求 (0字节payload)
   - 0x04: 配置命令 (变长)

4. 边界情况:
   - 帧头出现在payload中? SOF是2字节, 概率低
   - 数据校验失败? 丢弃整帧
   - 长度错误? 丢弃, 等待下一个SOF
   - 超时处理? 100ms无数据=断开

5. 并发考虑:
   - UART接收中断 + 主循环处理
   - 需要环形缓冲区
   - 帧解析状态机

6. 扩展性:
   - 保留命令范围: 0x80-0xFF用于应答
   - 版本字段? 先不加, 需要时扩展

7. 验证方案:
   - 单元测试: 各种边界帧
   - 压力测试: 连续发送1000帧
   - 故障注入: 随机字节, 校验错误

输出记录:
```markdown
## Sequential Thinking Record
**Trigger**: 设计协议帧格式
**Summary**: 固定帧头+变长payload+CRC8, 支持控制/心跳/遥测/配置
**Decision**: SOF=0xAA55, 状态机解析, 环形缓冲区
**Risks**: 帧同步丢失; 缓冲区溢出
**Verification**: 单元测试覆盖边界帧, 压力测试1000帧
```
```

## 常见错误

### ❌ 错误1: ST过度使用
```
场景: 修改一个注释
错误: 使用ST分析"是否应该修改这个注释"
正确: 直接修改, 不需要ST
```

### ❌ 错误2: ST记录不完整
```
场景: 使用了ST, 但只写了结论
错误:
## Sequential Thinking Record
Decision: 使用I2C1

正确:
## Sequential Thinking Record
**Trigger**: 配置新外设
**Summary**: I2C1配置已验证, 引脚/时钟/地址正确
**Decision**: 使用I2C1, PB6/PB7, 400kHz
**Risks**: 需测试总线繁忙检测
**Verification**: 逻辑分析仪验证波形
```

### ❌ 错误3: 跳过关键ST
```
场景: 配置电机PWM (安全关键)
错误: 直接写代码, 没有ST
正确: 必须使用ST分析:
      - PWM频率选择(为什么400Hz?)
      - 占空比范围(为什么1000-2000μs?)
      -  failsafe机制(失控后怎么办?)
```

## 豁免清单

以下情况**可以跳过**ST (但建议简短记录思考过程):

| 情况 | 示例 | 建议 |
|------|------|------|
| 纯格式调整 | 修改缩进、换行 | 无需ST |
| 注释修改 | 添加函数说明 | 无需ST |
| 变量重命名 | sensor_data → imu_data | 简单检查即可 |
| 复制已有配置 | I2C1配置复制到I2C2 | 3步快速验证 |
| 测试代码 | 临时调试打印 | 无需ST |
| 紧急修复 | 飞行器失控紧急补丁 | 先修复, 后补记录 |

## 故障排查

### 问题: 不确定是否需要ST

**解决方案**: 使用快速判断流程

```
                    开始
                      |
                      v
        +-----------------------------+
        | 是否涉及硬件配置?           |
        +-----------------------------+
           | yes              | no
           v                  v
    +-------------+    +------------------+
    | 使用ST(5-8步)|    | 是否安全关键?    |
    +-------------+    +------------------+
                          | yes        | no
                          v            v
                   +-------------+  +------------------+
                   | 使用ST(7-10)|  | 是否复杂算法?    |
                   +-------------+  +------------------+
                                     | yes        | no
                                     v            v
                              +-------------+  +-------------+
                              | 使用ST(8-10)|  | 无需ST      |
                              +-------------+  +-------------+
```

### 问题: ST过程太长, 影响效率

**解决方案**:
1. 检查是否在不必要的地方使用了ST
2. 降低思考深度 (如从10步减到5步)
3. 对于重复性任务, 建立模板后直接应用

### 问题: Reviewer认为不需要ST, 但Worker用了

**解决方案**: 这是可以的! ST是"保险", 多用比少用好。但要确保记录简洁。
