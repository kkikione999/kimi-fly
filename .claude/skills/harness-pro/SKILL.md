# Harness-Pro — System Overview

## Two Roles, One Clear Boundary

Harness-Pro has two distinct control agents. Understanding the boundary
between them is essential to the whole system working correctly.

```
┌─────────────────────────────────────────────────────┐
│  COORDINATOR (this skill)                           │
│                                                     │
│  Lives at: the skill level                          │
│  Knows about: phases, modules, overall progress     │
│  Decides: which phase is active, which module next  │
│  Does NOT: plan, write code, spawn subagents        │
│                                                     │
│  Talks to: hp-planner, Atlas, hp-health-check       │
└──────────────────────┬──────────────────────────────┘
                       │ approved plan / execution summary
┌──────────────────────▼──────────────────────────────┐
│  ATLAS (generated by hp-agent-factory)              │
│                                                     │
│  Lives at: the execution layer                      │
│  Knows about: the approved plan, task queue,        │
│               session learnings, current module     │
│  Decides: which subagent to spawn, parallel vs      │
│           sequential, how to handle failures        │
│  Does NOT: decide which module to work on,          │
│            communicate with the human, write plans  │
│                                                     │
│  Spawns: Explore, Librarian, Oracle, Backend,       │
│          Frontend, DBA, hp-refactor, hp-scribe,     │
│          hp-verifier, hp-health-check               │
└─────────────────────────────────────────────────────┘
```

**The contract between them:** Coordinator passes an approved plan file
to Atlas. Atlas executes it and returns an Execution Summary.
They do not communicate during execution — only at handoff points.
See `references/handoff-protocol.md` for the full contract.

## System Architecture

```
PLANNING LAYER
  hp-planner
  ├── Prometheus  (persistent — drives interview + plan writing)
  ├── Metis       (subagent mode — gap analysis, spawned once per plan)
  └── Momus       (subagent mode — accuracy review, spawned on request)
  Output → .harness/plans/{date}-{module}.md

            ↓ plan approved — Coordinator hands off to execution

BOOTSTRAP LAYER  (first time only)
  hp-cartography   → knowledge-hub/  (L0, L1, invariants, debt ledger)
  hp-agent-factory → .harness/agents/  (Atlas config + all subagent configs)
  hp-verifier      → baseline snapshot
  hp-health-check  → baseline score

            ↓ bootstrap complete — Coordinator starts execution cycles

EXECUTION LAYER  (repeating per module)
  Atlas (persistent per module lifecycle)
  └── spawns subagents per references/subagent-protocol.md:
      Explore, Librarian, Oracle, Backend-{lang}, Frontend, DBA
      → hp-refactor (writes code)
      → hp-scribe   (updates docs, same commit)
      → hp-verifier (validates)
      → hp-reviewer (reviews PR, auto-merges)
      → hp-health-check (re-scores)
  Output → Execution Summary → back to Coordinator

OBSERVATION LAYER  (continuous)
  .harness/observation-log.jsonl  ← machine-readable, for human dashboard
  .harness/alerts.md              ← structural alerts requiring human decision
  .harness/session-learnings.md   ← Atlas's accumulated wisdom per module
```

## Coordinator Flow

### Step 1: Triage

```
Determine:
- Primary language / framework
- Approximate scale (file count / line count)
- Most urgent pain points
- Solo or multi-engineer setup?
```

Output a **Triage Report** (max 20 lines) and decide:
- Is this a first-time setup? → go to Step 2 (Planning) then Step 3 (Bootstrap)
- Is this a continuation? → go to Step 4 (Execution Cycle)

### Step 2: Planning → hp-planner

Call `hp-planner`. Do not proceed until it outputs a file with
`status: approved` in `.harness/plans/`.

Coordinator does not participate in planning. It waits.

### Step 3: Bootstrap (first time per project)

```bash
gh repo clone {repo} && cd {repo} && git checkout main && git pull
```

Run in this order (2a and 2b in parallel):

| Step | Sub-Skill | Output |
|------|-----------|--------|
| 1 | hp-cartography | knowledge-hub/ complete |
| 2a | hp-agent-factory | .harness/agents/ complete |
| 2b | hp-health-check | baseline score recorded |
| 3 | hp-verifier | scaffold + baseline snapshot |

Bootstrap runs **once per project**. On subsequent modules, skip to Step 4.

### Step 4: Execution Cycle (per module)

```bash
git checkout -b harness/{module-name}-refactor
```

Hand the approved plan to Atlas. Atlas runs the full execution cycle
and returns an Execution Summary.

Coordinator reads the summary and decides one of:
- ✅ Module done → archive plan, start next module (back to Step 2)
- ⏭️ Module partially done, scope question → evaluate structural alert
- ❌ Module blocked → escalate to human observation layer

### Step 5: Post-Cycle

```bash
gh pr merge {pr-number} --squash --auto   # handled by hp-reviewer
git checkout main && git pull
git branch -d harness/{module-name}-refactor
```

## Human Role: Observer

Humans do not review PRs. They observe the system via:
- `.harness/observation-log.jsonl` — all score changes and events
- `.harness/alerts.md` — structural decisions requiring human input

**Intervention triggers (human acts on these, everything else is autonomous):**

| Alert | Example |
|-------|---------|
| Business invariant conflict | Two contradictory business rules |
| Architectural blocker | Circular dep fix needs scope the plan didn't authorize |
| Core dep unmaintained | Migration direction needs human decision |
| Health Score persistently declining | 3+ modules, score still falling |
| Reviewer rejecting repeatedly | Same module returned > 3 times |
| Momus cannot converge | Plan still REJECT after 3 iterations |
